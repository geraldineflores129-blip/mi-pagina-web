Código de Lenguaje de Máquina

Lexico

Análisis Léxico

Primera fase del compilador que convierte el código fuente en bruto en una secuencia de tokens, la unidad significativa más pequeña de un lenguaje de programación. El analizador léxico escanea el código carácter por carácter y agrupa los caracteres en tokens según las reglas del lenguaje.

Estos tokens pueden representar palabras clave, identificadores, constantes, operadores o símbolos de puntuación.

Ejemplo: int x = 10;

int  — keyword x  — identificador =  — operador 10  — literal numérico ;  — puntuación
Sintactico

Análisis Sintáctico (Parsing)

Fase que asegura que el código fuente sigue las reglas gramaticales del lenguaje de programación. Verifica que la secuencia de tokens producidos por el analizador léxico está correctamente organizada según la sintaxis del lenguaje.

Si el código no está estructurado según las reglas, el analizador genera mensajes de error apropiados.

Árbol de Análisis Sintáctico (Parse Tree)

Estructura similar a un árbol que representa la estructura sintáctica del código fuente. Muestra cómo se relacionan los tokens entre sí según las reglas gramaticales. Cada rama representa una regla de producción y las hojas representan los tokens.

Árbol Sintáctico Abstracto (AST)

Versión más abstracta del árbol de análisis. Representa la estructura jerárquica del código fuente con menos detalle, centrándose en la estructura sintáctica esencial. Es preferido en fases posteriores por ser más fácil de analizar y optimizar.

Papel del análisis sintáctico

  • Toma tokens producidos por el analizador léxico como entrada
  • Verifica la estructura gramatical del programa
  • Detecta errores de sintaxis (puntos y coma faltantes, paréntesis sin cerrar…)
  • Construye un árbol de análisis o AST para fases posteriores
Semántico

Análisis Semántico

Garantiza que el código fuente tenga un significado lógico. Mientras el análisis sintáctico comprueba la estructura, el análisis semántico verifica el significado.

Esta fase identifica errores como desajustes de tipos, variables no declaradas y operaciones inválidas.

Comprobaciones clave

  • Comprobación de tipos: garantiza operaciones sobre tipos compatibles. Intentar sumar una cadena y un entero se marcaría como error.
  • Declaración de variables: comprueba que las variables se declaren antes de usarse.

Ejemplo de error semántico

int a = 5;
float b = 3.5;
a = a + b;  // Error: no se puede asignar float a int
Intermedio

Generación de Código Intermedio

El código fuente se convierte en una Representación Intermedia (IR), situada entre el código fuente de alto nivel y el código máquina final. No está hecho para ninguna máquina específica, lo que facilita la portabilidad y optimización.

Actúa como un puente entre el front end (fases de análisis) y el back end (generación de código) del compilador.

Importancia

  • Independencia de plataforma: la misma IR puede reutilizarse; solo la generación de código varía por plataforma.
  • Eliminación de código muerto: partes que no afectan la salida.
  • Optimización de bucles: mayor velocidad y menor uso de memoria.
  • Eliminación de subexpresiones comunes: reutilizar valores calculados.

Ejemplo: a = b + c * d

T1 = c * d
T2 = b + T1
a  = T2
Optimización

Optimización del Código

Proceso de mejorar el código intermedio o destino para que el programa se ejecute más rápido, use menos memoria o sea más eficiente, sin alterar su funcionalidad.

Técnicas comunes

  • Plegamiento constante: pre-cómputo de expresiones constantes.
  • Eliminación de código muerto: eliminación de código inaccesible o no utilizado.
  • Optimización de bucles: movimiento invariante del código o desenrollamiento.
  • Reducción de fuerza: sustituir operaciones costosas por otras más sencillas.

Antes

for (int j = 0; j < n; j++) {
  x = y + z;
  a[j] = 6 * j;
}

Después

x = y + z;
for (int j = 0; j < n; j++) {
  a[j] = 6 * j;
}

Generación de Código

Fase final donde la representación intermedia se traduce en código máquina o ensamblador. El código objeto resultante debe tener el significado exacto del código fuente y ser eficiente en uso de CPU y gestión de memoria.

Detalle

Análisis Sintáctico — En Profundidad

1. Gramáticas Libres de Contexto (CFG)

Una CFG define las reglas sintácticas de un lenguaje mediante reglas de producción que describen cómo se forman cadenas válidas. Son suficientemente potentes para representar:

  • Estructuras anidadas
  • Paréntesis equilibrados
  • Constructos de programación estructurada por bloques

2. Derivaciones

Proceso paso a paso de aplicar reglas gramaticales para partir del símbolo inicial y producir una cadena válida de tokens. Ayuda al analizador a construir el árbol de análisis.

3. Árbol de Sintaxis Concreta (CST)

Representa la estructura gramatical completa del código fuente, incluyendo todos los símbolos gramaticales y la puntuación.

4. Árbol de Sintaxis Abstracta (AST)

Versión simplificada del CST que elimina detalles innecesarios y mantiene solo la estructura esencial. Preferido en fases posteriores por ser más fácil de analizar y optimizar.

5. Ambigüedad

Una gramática es ambigua cuando la misma cadena puede analizarse de más de una manera, produciendo múltiples árboles. Esto puede confundir al compilador, por lo que los lenguajes se diseñan usando gramáticas no ambiguas.

Análisis descendente y ascendente

Top-Down (Descendente)

Construye el árbol desde la raíz hacia las hojas. Incluye: Análisis Recursivo Descendente y Análisis LL.

Bottom-Up (Ascendente)

Construye el árbol desde las hojas hacia la raíz. Incluye: Reducción-Desplazamiento y Análisis LR (LR(0), SLR, LALR, CLR).

Avanzado

Traducción Dirigida por Sintaxis (SDT)

Método que integra el análisis sintáctico con reglas semánticas para producir código intermedio, código máquina o instrucciones optimizadas. Cada regla gramatical está vinculada a acciones semánticas que definen cómo debe producirse la traducción.

Estas acciones ayudan a evaluar expresiones, comprobar tipos, generar código y gestionar errores.

Tipos de Atributos

Sintetizados

  • Se calculan usando los valores de los hijos y el propio nodo.
  • Evaluados en recorrido bottom-up.
  • Pueden asociarse a terminales y no terminales.
  • Usados en SDT de tipo S y L.

Heredados

  • Se calculan usando valores del padre, hermanos y el nodo.
  • Evaluados en recorrido top-down / lateral.
  • Solo se asocian a no terminales.
  • Usados solo en SDT de tipo L.

Ejemplo Práctico: S = 2 + 3 * 4

Evaluación bottom-up (de hojas a raíz)

  1. F → 2  →  F.val = 2
  2. F → 3  →  F.val = 3
  3. F → 4  →  F.val = 4
  4. T → F  →  T.val = 3
  5. T → T * F  →  T.val = 3 × 4 = 12
  6. E → T  →  E.val = 12
  7. E → E + T  →  E.val = 2 + 12 = 14

Ventajas y Limitaciones

✓ Ventajas

  • Sencillo de diseñar e implementar
  • Estructura modular clara
  • Traducción eficiente con soporte de optimizaciones básicas

✗ Limitaciones

  • Capacidad limitada de traducción
  • No adecuado para traducciones complejas
  • Manejo de errores débil
Auxiliar

Tabla de Símbolos

Estructura de datos utilizada y mantenida por el compilador que contiene todos los nombres de identificadores junto con sus tipos. Permite al compilador encontrar identificadores rápidamente durante el proceso de compilación.

Durante el análisis sintáctico, información sobre identificadores —como nombres y ámbitos— se registra y actualiza en la tabla de símbolos para su uso en fases posteriores.

Tema 01

Conceptos Básicos

Componente léxico

Token

Elemento léxico del lenguaje. Símbolo No Terminal de las fases siguientes. Representa la unidad mínima con significado en el código fuente.

Expresión regular

Patrón

Expresión regular que define el lenguaje. Forma general: Letra (Letra | Dígito)*. Describe el conjunto de lexemas que puede tomar un token.

Cadena de caracteres

Lexema

Cadena de caracteres correspondiente al componente léxico. Es la representación concreta (texto real) del token en el código fuente.

Metadatos

Atributos de un Componente

Información del componente necesaria en etapas posteriores del análisis: valor de la constante, nombre de una variable, etc.

Categoría Léxica

Tipo de símbolo elemental del lenguaje fuente. Ejemplos: identificadores, palabras clave, constantes numéricas, operadores, etc. El token es el elemento perteneciente a una categoría léxica.

Función del Analizador Léxico

Función principal

El analizador léxico es la primera fase de un compilador. Su función principal consiste en leer los caracteres de entrada y elaborar como salida una secuencia de componentes léxicos que utiliza el analizador sintáctico para hacer el análisis.

Lee la secuencia de caracteres de izquierda a derecha del programa fuente y agrupa la secuencia de caracteres en unidades con significado propio (componentes léxicos o tokens). Las palabras clave, identificadores, operadores, constantes numéricas, signos de puntuación como separadores de sentencia, llaves, paréntesis, etcétera, son diversas clasificaciones de componentes léxicos.

Razones para separar el Análisis Léxico del Sintáctico

1

Diseño más sencillo. Separar el análisis léxico del análisis sintáctico a menudo permite simplificar una u otra de dichas fases, resultando en un diseño general más modular y mantenible.

2

Mayor eficiencia del compilador. Un analizador léxico independiente permite construir un procesador especializado y potencialmente más eficiente. Con técnicas especializadas de manejo de buffer para la lectura de caracteres y procesamiento de componentes léxicos se puede mejorar significativamente el rendimiento.

3

Mejor transportabilidad. Las peculiaridades del alfabeto de entrada y otras anomalías propias de los dispositivos pueden limitarse al analizador léxico, facilitando la portabilidad del compilador.

Análisis Léxico — Scanner

Scanner tiene las funciones de leer el programa fuente como un archivo de caracteres y dividirlo en tokens. Los tokens son las palabras reservadas de un lenguaje, secuencia de caracteres que representa una unidad de información en el programa fuente.

En cada caso un token representa un cierto patrón de caracteres que el analizador léxico reconoce desde el inicio de los caracteres de entrada. Es necesario generar un mecanismo computacional que permita identificar el patrón de transición entre los caracteres de entrada, generando tokens que posteriormente serán clasificados. Este mecanismo es posible crearlo a partir de un tipo específico de máquina de estados llamado autómata finito.

Para reconocer

Los lexemas de un lenguaje se reconocen usando expresiones regulares; para reconocer estructuras sintácticas se usan gramáticas independientes de contexto.

Tema 03

Análisis Sintáctico

Determina si la secuencia de componentes léxicos sigue la sintaxis del lenguaje y obtiene la estructura jerárquica del programa en forma de árbol, donde los nodos son las construcciones de alto nivel del lenguaje.

Se determinan las relaciones estructurales entre los componentes léxicos, algo semejante a realizar el análisis gramatical en el lenguaje natural. La estructura sintáctica se define mediante las gramáticas independientes del contexto.

¿Qué es el Analizador Sintáctico?

Es la fase del analizador que se encarga de chequear el texto de entrada en base a una gramática dada. Si el programa de entrada es válido, suministra el árbol sintáctico que lo reconoce. En la práctica, el analizador sintáctico también realiza:

Acceder a la tabla de símbolos
Chequeo de tipos (análisis semántico)
Generar código intermedio
Generar errores cuando se producen

Tema 04

Análisis Semántico

El análisis semántico dota de significado coherente a lo construido en el análisis sintáctico. El chequeo semántico se encarga de que los tipos que intervienen en las expresiones sean compatibles o que los parámetros reales de una función sean coherentes con los parámetros formales.

Funciones Principales

Identificar cada tipo de instrucción
Completar la Tabla de Símbolos
Comprobaciones de tipos
Comprobaciones del flujo de control
Comprobaciones de unicidad
Comprobaciones de emparejamiento

La tarea del compilador requiere la extracción del contenido semántico incluido en las distintas sentencias del programa. Es necesario dotar al compilador de rutinas auxiliares que permitan captar todo aquello que no se ha expresado mediante la sintaxis del lenguaje.

Verificación de Tipos

Un componente importante del análisis semántico es la Verificación de Tipos. El compilador verifica si cada operador tiene operandos permitidos por la especificación del lenguaje fuente. La fase de análisis semántico revisa el programa fuente para encontrar errores semánticos y reúne la información sobre los tipos para la fase posterior de generación de código.

El análisis semántico completa las dos fases anteriores (lexicográfico y sintáctico) incorporando ciertas comprobaciones que no pueden asimilarse al mero reconocimiento de una cadena dentro de un lenguaje.

Componentes auxiliares del compilador

  • Tabla de Símbolos: utilizada por todas las fases del compilador para almacenar y consultar información sobre identificadores.
  • Manejador de Errores: detecta, reporta y se recupera de errores en cada fase sin detener todo el proceso de compilación.
Tema 05

Representación de Operaciones y Generación de Código

La representación de operaciones dentro de un compilador consiste en modelar las expresiones y cálculos del programa mediante estructuras internas que permitan analizarlas, optimizarlas y posteriormente traducirlas a código máquina.

Representación de Operaciones

Operación

Es una acción que transforma uno o más operandos mediante un operador.

Operandos y Operadores

Los operandos representan los datos y los operadores indican la acción que debe realizarse sobre ellos.

Operaciones Simples

Contienen un único operador, por ejemplo: a + b.

Operaciones Compuestas

Combinan varias operaciones en una misma expresión, como a + b * c.

Importancia

Una representación correcta permite optimizar expresiones, eliminar cálculos redundantes y generar código más eficiente.

Flujo de Datos y Flujo de Control

Concepto Descripción
Flujo de Datos Describe cómo los datos se mueven y transforman dentro del programa.
Flujo de Control Describe el orden en que se ejecutan las instrucciones.
Condicionales Introducen bifurcaciones y caminos alternativos de ejecución.

Representaciones Intermedias

Una representación intermedia (IR) es una forma de código independiente de la arquitectura utilizada entre el análisis y la generación de código final.

Características

  • Independiente de la máquina.
  • Fácil de optimizar.
  • Facilita la portabilidad.
  • Representa claramente las operaciones.

Código de Tres Direcciones

t1 = b * c
t2 = a + t1
x = t2

El código de tres direcciones divide expresiones complejas en operaciones simples usando variables temporales.

Árboles Sintácticos como Representación Intermedia

Los árboles sintácticos muestran la jerarquía de las operaciones y permiten representar la precedencia de operadores de forma natural.

Variables Temporales

Son variables creadas por el compilador para almacenar resultados intermedios durante la evaluación de expresiones. 

t1 = b * c
t2 = a + t1
resultado = t2

Generación de Código Final

La generación de código final traduce la representación intermedia a código máquina o ensamblador específico de la arquitectura destino.

Algoritmo de Sethi-Ullman

El algoritmo de Sethi-Ullman determina el orden óptimo para evaluar expresiones y minimizar el número de registros necesarios durante la generación de código.

Ventaja

Reduce accesos innecesarios a memoria y mejora el uso de registros.

Pila de Ejecución

La pila almacena parámetros, variables locales y direcciones de retorno durante las llamadas a funciones.

Optimización del Código Objeto

  • Eliminación de código muerto.
  • Plegamiento de constantes.
  • Propagación de copias.
  • Optimización Peephole.
  • Simplificaciones algebraicas.
